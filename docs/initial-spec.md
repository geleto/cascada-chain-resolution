I am implementing a programming language that uses copy-on-write for variable assignment and modification.
We support objects and arrays, the language is compiled into JavaScript.
But whenever possible we shall still reuse the structure.
When modifying only a child property - we shallow copies the object at each level along the modified path, iterating deeper only along the path and finally setting the changed property.


// Illustrative code to show shall-on-each-path-level copying
function simpleAssignPath(root, segments, value) {
  // No segments left: replace the whole value.
  if (segments.length === 0) {
    return value;
  }

  const [head, ...tail] = segments;

  if (root === null || typeof root !== "object") {
    return { [head]: simpleAssignPath(null, tail, value) };
  }

  const newChild = simpleAssignPath(root[head], tail, value);

  if (Array.isArray(root)) {
    const newRoot = [...root];
    newRoot[head] = newChild;
    return newRoot;
  }
  return { ...root, [head]: newChild };
}

We need to be selective when mutating the object. We use copy-on-write, but only when the object has moved outside of the variable (assigned to other vars, used as arguments, etc…).
var a = { x:1, y:2 }
a.x = 2 // no copy on write
var b = a;
a.x = 1 // this requires copy on write to avoid changing b

The same principle works selectively on branches

var location = { pos: {x:1,y:2}, delta = {x:3, y:4} }
var pos = location.pos
location.pos.x = 2 // copy on write of pos to avoid changing it
location.delta.x = 5 // no copy on write

Objects imported from outside calls also need not be modified, so copy on write is used on them too.

Additionally, promises are handled transparently. If a property is set to a promise, once that promise is resolved, it is replaced by the resolved value automatically. If the value rejects, it is replaced by the language error value. Cascada's real error value is a special immediately-rejecting thenable, so rejecting promises are valid error values too; this sandbox often uses JavaScript `Error` objects to stand in for that value. Below, `Error` means this language-level error value unless JavaScript `Error` is mentioned explicitly. Also assignPath can receive an error value as value and it has to be properly referenced(see Subtree counters). Any operation or walk that encounters a settled-but-unreplaced promise replaces it in place (with counter propagation), and registers writeback for pending unregistered ones it discovers.

We have these operations/helpers:
import(obj) - for marking outside objects shared. Always walks obj: validates cycles and the frozen rule at the boundary, mints a promise mirror for every promise-valued key, and marks resolved promise values shared (recursively, in import mode). It does not build counter metadata - indexing is lazy and branch-level (see Subtree counters). (The old rescan=false option is removed - the marking, mirror, and validation obligations cannot be skipped.) Imported Error values remain language errors and count for hasError/normalize (see Subtree counters). Works with promises - it is marked as shared after resolve. External objects are not expected to be mutated by outside code.
assignPath(root, segments, value) - for changing the whole object or any path value. Any branch/property with the shared mark will be copied-on-write. Importing objects with promises is not recommended as the imported object will have the promise rewritten to the resolved value. For a data chain, missing/null/undefined intermediate path segments are created as blank objects; assigning through an existing primitive/untracked intermediate turns that property into an Error. Path segments are objects, properties are assigned. An Error at the target key is replaceable; an Error used as an intermediate path node is not entered.
deletePath(root, segments) - for deleting a path. Empty path deletes/replaces the whole root and returns null. Can not be used on arrays. It is of course possible to use transformation that creates a new array without the element, but this is done outside the variables. An Error at the target key is deletable; an Error used as an intermediate path node is not entered.
lookupPath(root, segments, sharedOwnership=true) - returns the value. By default the return is marked as shared, because the value is now shared between its owning variable/path and the caller; because of this even root access has to go through lookupPath. If sharedOwnership is false, the lookup does not create shared ownership, so the returned object/array is not marked. This is valid for pure read/inspection, or when ownership is ceded to the caller as the last use of the original owner.
normalize(root, segments, full=false) - resolves the path to the target branch with the same path rules as lookupPath (a promise along the way is continued through its mirror; an Error along the path yields an Error; empty segments target the whole variable), marks the reached branch shared at call time, waits for the promises inside it to settle, and hands out the branch's current state with all promises resolved - not an end-of-all-activity snapshot of the future. The call-time mark is what makes "current state" exact: later operations copy-on-write away from the marked branch, so it can only settle, while suspended remainders of earlier operations still land in it through in-place mirror advances. (The return escapes to the caller, so the mark was owed in any case.) Set full=true to perform a complete copy with no marks left. If there is any Error in the reached branch - the return is a single Error (just a single error indicating possibly many - this is a sandbox test implementation). Returns a promise that resolves to a complete value with no promises/errors in it, or an Error object if any Error or rejecting promise was still left.
hasError(root, segments) - whether the property is an error, or there was an error along the path. Resolves the parent path (a broken path - intermediate Error, missing, or primitive - is true; a missing terminal property is false), then answers for the current snapshot, guided by the counters: errorCount > 0 - true immediately; pendingCount 0 - false immediately; otherwise it follows only the pending promises (descending only nodes with promiseCount > 0) and probes again at each settlement - first error wins, false once everything settles. A pure query: it never marks anything.

All of these run immediately - hasError and normalize too: their synchronous prefix indexes the branch and registers at their program position, and they are simply the operations whose returned promise the program may await; we do not await one operation before issuing the next. One thing to keep in mind - the first to await will be the first to continue execution, so operations will run in order despite awaiting the same promises, there will be no race conditions. All operations (the actual calls: assignPath, lookupPath, normalize/etc…) to the same root shall happen sequentially - this is enforced by the implementer. Sequentially means issued and committed in program order, not awaited to completion: each operation's synchronous prefix runs to the end - committing its synchronous changes and registering its continuations at its own program position - before the next operation is issued. Suspended remainders complete later, in that same order per promise (FIFO). No operation is ever awaited before the next one is issued - that would serialize the program and make the FIFO machinery pointless.

Sharing is evaluated only from the point of the target language. Promises and resolved values are interchangeable (promises are implicitly resolved), none of the markings are visible, etc… The shared mark is a field of the runtime's per-node metadata record, a single non-enumerable Symbol property ignored by the language (see Subtree counters - Metadata record). Non-extensible external objects, such as frozen/sealed objects, are treated as already safe for COW without adding any metadata.

A promise exposed through shared-ownership lookupPath/import carries the same SHARED obligation as a directly returned object/array. If it resolves to an object/array, the resolved value is marked SHARED before consumers can observe it.

Shallow Copy - a COW copy duplicates the language-visible enumerable string keys of the object/array into a fresh object/array, preserving array length. As the object that requires such copying is shared, the copy is the new owned/unmarked node. The source object keeps any shared mark it already has; marks are never removed. Reused non-path child objects/arrays copied from a shared branch are also marked, so shared references stay protected. The current path child is tracked by the walk's inherited shared state and becomes owned/unmarked after it is replaced or copied for the current mutation, unless the assigned/reused object already carried its own shared mark. Another, simpler approach is to deep-freeze the whole copied branch. And a third approach is to add both SHARED and NOT_SHARED markings, and using not-shared for the new along-the-path properties. The copy never inherits the source's metadata record - it gets a freshly constructed one (owned/unmarked, fresh mirror map, counter snapshot; see Subtree counters - commit points, shallowCopy). Every copied key that holds a promise is forked eagerly at copy time (see Value resolve - the Fork birth point); non-path promise captures are marked shared, while the current path promise is protected by inherited shared state when the walk advances through it. The scan is O(width of the node), the same order of cost as the copy itself.

Shared Marking (requiring copy-on-write) - All branches that are shared are marked at the root. The mechanisms to mark an object shared are 2 - import and shared-ownership lookupPath. Imported data need not be modified and this is the only semantic state import adds - mark; the import boundary scan also registers promise mirrors/writebacks for promise-valued descendants. If an object is non-extensible, the runtime treats that as a shared mark instead of writing one. When we have a shared-ownership lookupPath at a sub-branch - it has to be marked as shared too. Nested shared marks are fine. A shared mark logically also applies to children, but they need not carry it. Mutating walks therefore keep track of whether the current path node is inside a shared branch, even if the node itself has no direct SHARED mark. All operations that pass/return/store a whole existing object or branch must use sharedOwnership=true unless they can prove ownership is not shared; pure inspections and final ownership transfer may use sharedOwnership=false.

Value resolve - each node can have a hidden promise mirror map, held in the per-node metadata record (see Subtree counters - Metadata record), created for every key that was assigned a promise. (Inline metadata, not a WeakMap side table: the frozen rule guarantees non-extensible nodes never hold promise keys, so the inline record always suffices.) Each promise mirror stores {promise, currentValue}: the exact promise assigned to that key, and the newest version of its resolved value. `currentValue` is changed by each operation that mutates it, in operation order, so each op's change is applied on top of the previous `currentValue` rather than on the original resolved value. A normal owned promise writeback does not mark the resolved object/array shared; the resolved value is still owned by the single Cascada location that received the promise. Import and shared-ownership lookupPath mark resolved objects/arrays shared because they cross an ownership boundary. Forked non-path promise mirrors also mark the captured value because a copied world and the source world initially reuse that same `currentValue`; a forked current-path promise instead COWs from the walk's inherited shared state if the walk advances through it. The resolved value is written back into its property only if its promise mirror is still the one the map holds for that key. If a later op has since reassigned the key (even to the same promise) - a new promise mirror is created. Even if not written to the node property - we keep `currentValue` alive privately - for reads that captured the mirror - without touching the node property. Every synchronous overwrite/delete through the runtime must clear the promise mirror map for that key, even when assigning the same value. All required accessible promises must start resolving immediately: we use .then and continue walking rather than await + enter, leaving other immediately accessible promises to resolve in parallel (and yet in operation order as guaranteed by JS).

A promise mirror is born at exactly three points, and where its starting `currentValue` comes from differs:

- Assignment: a promise is assigned to a property (assignPath value, literal initialization). A fresh promise mirror is always created, even if the same promise instance was already at this key - two assignments are divergent worlds and must not share `currentValue`. The writeback continuation is registered immediately at assignment, before any operation can register on that promise for this key, so the writeback is always the first responder and `currentValue` starts as the raw resolved value.

- Discovery: a walk or operation reaches a promise key that has no mirror (imported data, raw literals - any orphan). The mirror is created and its writeback registered at the discovering operation's program position; `currentValue` starts as the raw resolved value.

- Fork: a copy-on-write shallow copy duplicates a key that holds a promise. The copy's promise mirror must be created eagerly, at copy time - never minted lazily by a later walk. A lazily minted mirror would seed `currentValue` from the raw resolved value, silently losing every mutation made by operations issued before the copy. At copy time, for each copied key that holds a promise: ensure the source node has a mirror (Discovery, if it is an orphan), then create the copy's mirror and register its initializer immediately, at the copier's program position. This initializer never reads the raw resolved value - it reads the source mirror's `currentValue` at its own FIFO slot and uses it as the copy's starting `currentValue`, mirroring it into the copy's property only if the copy's key still holds the promise unchanged. Non-path captures are marked shared because they are now reused by both worlds; the current path capture is left unmarked and relies on the walk's inherited shared state if the mutation continues through it. Because the initializer is registered at the copier's program position, it observes exactly the advances of operations issued before the copy and none issued after - the two worlds diverge at precisely the copier's place in program order, uniformly for pending and settled-but-unreplaced promises. After the fork the mirrors are independent: operations traversing the copy advance the copy's mirror, operations traversing the old node advance the source mirror, and either a shared mark on the captured value or inherited shared state forces the first advance that would mutate a reused value to copy. Value advances themselves go through the same shallow copy, so a resolved value that contains nested promise keys is forked by this same rule automatically.

assignPath - If it reaches a promise in the intermediate path it resolves it and continues through the mirror's `currentValue` (see Value resolve), COWing that value and advancing the mirror. If an intermediate path segment is missing/null/undefined, create it (a blank {}). If an existing intermediate property is primitive/untracked, replace that property with an Error and stop. Continue through the path (resolving any promises along the way), assigning the value when the path end is reached. Because it is the first to await the promise, it is the first to resume and modify. When it reaches an object marked shared along the path it copies it (the COW approach above). assignPath into an intermediate error node is a no-op, while assignment at an Error target replaces it. It installs the copied spine top-down as it descends, so the new root identity is decided synchronously and rebinding is synchronous; the variable must be associated with the returned root before the next command is issued. The return may be a synchronous new or old root, or a promise for the new root only when the root itself is pending, or an Error when the root is already an Error. A returned rejecting promise is allowed; in Cascada it behaves like the language error thenable, while this sandbox may still convert rejections to `Error` objects inside property/mirror continuations.

lookupPath - returns the value it reaches. If that value is a promise, or if a promise lies along the path, it returns a promise that resolves to the value once reachable - continuing through `currentValue` if the path was mutated during resolution (see Value resolve), never the raw resolved value. The returned promise is not awaited by lookupPath itself; values resolve only at the point of consumption. If sharedOwnership is true, marks the return shared if it is an array or object; if the return is a promise, it ensures the resolved value is marked shared. sharedOwnership=false lookups are for inspection or final ownership transfer and do not share ownership. Return semantics otherwise as assignPath. Any Error or rejecting promise along the path results in an Error return.

deletePath - resolves the intermediate path segments to reach the target (a segment may be a promise, continued through `currentValue` per Value resolve, not awaited), then deletes it, clearing the key's promise mirror so no later writeback re-mirrors it. COWs shared branches along the way (the COW approach above). Objects only; array element deletion is a no-op in this helper. Any intermediate Error along the path is a no-op, while an Error target can be deleted. Returns the new root, with the same synchronous/pending/Error semantics as assignPath.

Optimizations:

Subtree counters (implementation - supersedes the CLEAN flag, and settles the RefCounting question: we count promise/error content, not references; ownership stays mark-based). Every indexed node carries exact counts of the pending promises and Error values reachable in its subtree (indexing is lazy and branch-level - see below). hasError becomes O(1) at a settled node, and normalize's completion becomes an event - the watched pendingCount reaching 0 - instead of repeated rescanning. CLEAN survives only as a derived property: clean ⇔ pendingCount + errorCount === 0. Counters answer "what is pending/broken below me"; the SHARED mark answers "who else can see me" - the two never substitute for each other.

Metadata record - all runtime metadata lives in one per-node record under a single non-enumerable Symbol, ignored by the language and never copied by the shallow copy. The SHARED flag and the promise mirror map share this record with the counters rather than owning separate Symbol properties. Under lazy branch-level indexing, record existence does not imply counted-ness: mirrors and shared marks can create a record before counters are live.
- shared - the SHARED mark; set once, never cleared; false at birth is the old "no mark". hasSharedMark(v) = meta(v)?.shared === true || !Object.isExtensible(v).
- mirrors - the promise mirror map (see Value resolve), lazy.
- pendingCount, errorCount - the counters, valid only when the node is indexed. pendingCount: promise-valued own keys contribute 1 each; each tracked child edge contributes the child's pendingCount times the edge multiplicity. errorCount: the same for Error values.
- parents - exact reverse edges and the indexed marker. `parents === undefined` means counters are not live; an empty `Map` means indexed root / no indexed parents; a populated `Map` is `Map<parentNode, edgeCount>` (a parent referencing the same child under two keys has multiplicity 2). Every commit that creates or removes a parent/child edge updates it; when an edge count reaches 0, delete that parent entry, never the `parents` map itself. References are strong in the sandbox (a world held by a suspended reader must keep receiving deltas); WeakRef edges pruned on dead deref are required in the real implementation, sound because liveness then coincides with observability - with strong edges every COW'd-away world stays referenced by its reused children and keeps receiving deltas, so per-delta cost grows with the COW history of a hot key: a time tax, not just memory.
- settlementWatchers - optional array added only by normalize/hasError while callers are waiting for pendingCount to verify at 0.
- settlementVerifyScheduled - boolean latch that coalesces settlement verification callbacks while settlementWatchers exists.

Frozen rule: non-extensible (frozen/sealed) objects are forbidden to contain promises or Error values anywhere beneath them - validated by boundary screening/indexing, a violation commits an Error at the written key or returns an Error from import/normalize. A non-extensible node cannot be mutated in place, permanently counts as (0,0), and carries no metadata record at all: no counters, no parents, no promise mirrors. This keeps all metadata in the one inline Symbol property; no out-of-band side table exists.

countsOf(value): pending or settled-but-unreplaced promise (1,0); Error (0,1); non-extensible node (0,0); tracked node its stored totals (indexing first if `parents === undefined` - tested via the `parents` field, never via record presence); primitive (0,0). Counts are purely structural - a function of what is physically at the keys - and change only at commit points, in the same FIFO slots that mutate the tree, so they inherit the ordering guarantees.

applyCountDelta(node, dPending, dError): add to the node's counters, then recurse into every parent multiplied by edge count; zero deltas short-circuit. Propagation termination relies on acyclicity, which is thereby promoted from an optimization assumption to a hard invariant (see Cycles).

Lazy, branch-level indexing - counter metadata exists only where normalize/hasError have been used; everything else pays zero bookkeeping (mirrors and shared marks are independent of indexing and always maintained). The first normalize/hasError on a branch indexes exactly that branch, at the caller's program position; indexing stops at sub-branches that already carry counter metadata, registering the boundary edge and reusing their totals; from then on the indexed region is maintained forever. Deltas stop naturally at a region's top (no parents above); indexing an ancestor branch later connects the regions through the boundary edge. Two load-bearing rules: (1) indexed regions are downward-closed - indexing walks whole subtrees and a write into an indexed parent indexes the entering value; every ancestor chain is then an unindexed prefix followed by an indexed suffix, and a write-created cycle must return to the written target, so validation's descent catches it. (2) The indexed gate is evaluated at commit time, never captured at registration: a continuation registered while a node was unindexed performs full bookkeeping if the node is indexed when it commits; overlapping indexing attempts over the same pending promise are safe by idempotence (`parents` is created only once, and the mirror map reuses the same mirror for the same promise - no duplicate writebacks or marks). Copies inherit the source's indexing. Boundary validation is eager even where indexing is lazy: import and every external writeback validate the frozen and cycle rules the moment data enters - only counter building is deferred - so the language invariants (acyclicity, frozen purity) hold at all times, not merely in indexed regions. For a writeback target, validation descends the value checking node === target - sufficient, because a write-created cycle must reach the written target. An indexing-time violation (defensive) surfaces by language semantics: normalize returns the Error, hasError returns true - only the write helpers commit an Error at a key. Cost: never-checked data pays zero; the first check pays one O(branch) scan - unavoidable, something must find the promises once; repeated checks are O(1).

Indexing / boundary screening - counter indexing initializes counters bottom-up, registers parent edges, and mints a promise mirror (Discovery) for every promise-valued key in an indexed region. Eager minting is required because nothing rescans indexed regions: an orphan promise with no registered writeback would hold pendingCount up forever. Enforces the frozen rule. Transactional - two passes: a pure validate pass (cycle coloring, frozen rule, direct target reachability for writebacks; no metadata, no mirrors, no edges) followed by a commit pass that cannot fail (metadata, counters, mirrors, parent edges) - single-pass minting would leave live DAG-shared nodes carrying parent edges into a rejected value when a late violation aborts it. Import runs boundary screening/marking mode: every promise key gets a continuation that marks its resolved value shared and re-screens it in the same mode, but import builds no counters. Cycle checking lives in boundary validation/indexing: cycles can enter solely through external data - internal operations cannot create them, because COW copies the target before a self-referencing set (a.property = a installs the pre-copy object into the new copy). Cycle guard for externally-resolved values: if the value reaches the write target, commit an Error instead - this closes the writeback back-edge, the one way a cycle could enter the graph. The back-edge check is one universal mechanism: validation descends the value checking node === target before any early exit - sufficient for any target, because a write-created cycle must reach the written target. (The ancestor closure - stop early at indexed nodes, computed lazily - is an optimization for indexed targets, added only if profiling demands it.) Two separate entry points - validation and indexing are distinct concerns. Eager: boundary validation (import and every external writeback - frozen and cycle rules checked the moment data enters; import additionally mints mirrors and registers mark-on-settle continuations, building no counters). Lazy: indexing (the first normalize/hasError on a branch - the region creator, at the caller's program position; a write into an already-indexed parent; the commit half after writeback validation when the target is indexed). Traversal uses two-color marking, not a flat seen set: visiting (on the current descent stack) and visited (completed). Reaching a visiting node is a cycle and commits an Error; reaching a visited node is a DAG share - register the edge and reuse its totals without recursing.

Commit points - every runtime property write or delete goes through two small helpers, setProperty(parent, key, value) and deleteProperty(parent, key), which own exactly the counter bookkeeping: index an entering value when needed, delta at the written node (new property → +countsOf(new); replaced property → −countsOf(old) +countsOf(new); delete → −countsOf(old), the cleared key's contribution simply removed), parent-edge swap, propagation. No bare property write or delete exists outside them. Promise mirrors are deliberately not the helpers' concern: mirror lifecycle (fresh mirror on promise assign, guarded writeback, clear on overwrite/delete) stays at the operation sites as in the base kernel - the helpers need no mirror awareness because a promise physically at the key already counts (1,0) through countsOf, so assign/writeback/delete deltas come out right automatically. The caller sites:
assignPath target - overwrite; a newly assigned pending promise contributes (1,0), its mirror registered at assignment as before.
deletePath target - subtract the removed value's counts, remove the edge.
Promise writeback (mirror still owns the key) - replace (1,0) with countsOf(resolved value), indexing it if needed. A revoked writeback commits nothing to the live world; a suspended commit into a disconnected branch runs the same code and propagates only through that world's ancestors - exact per world.
Walk installs - the synchronous next !== child replacement and the suspended parent[key] = next, same rule.
shallowCopy - the copy copies language-visible keys and forks promise mirrors as today, then calls `copyCounters(source, copy)`. That hook is a no-op unless the source is indexed; indexed copies snapshot counter totals, receive `parents = new Map()` (indexed copy, no indexed parents yet), and register themselves as a parent on every reused tracked child.

normalize(root, segments, full) - the path resolves like lookupPath; the reached branch is then marked shared at normalize's program position - the mark pins the snapshot; completion is count-driven. normalize and hasError are ordered like every other operation: issued at their program position, their synchronous prefix indexes the branch (activating the counter metadata) and registers on its pending promises there, and their remainders resume through the uniform wrapper in FIFO order, interleaving correctly with other operations' continuations on the same promises. The runtime never schedules outside the wrapper. Completion is branch settlement (pendingCount reaching 0) - exact because of the call-time mark: later-issued operations COW away from the marked branch and cannot add promises to it, while suspended remainders of earlier-issued operations still land in it through in-place mirror advances; the wait-set is precisely the promises present at the call plus, recursively, promises arriving inside their resolved values. No early exit on errors: an earlier-issued remainder can still replace an error before settlement, so collapse-to-Error is decided only at zero. Both ops build and thereafter maintain the counter metadata - the first call on a branch pays the indexing scan, later calls are O(1) plus path resolution. hasError never sets the SHARED mark - a pure query must not change ownership state; normalize marks at call time because its return escapes anyway, and the early mark is what makes the snapshot semantics true. whenSettled(node) - used by normalize only: optional `meta.settlementWatchers` is created only while normalize callers are waiting; `meta.settlementVerifyScheduled` only coalesces the queued check. Verification is registered through the uniform wrapper, on the promise whose settlement caused the zero-crossing, landing at the tail of that promise's FIFO queue - only while waiters exist, both when a commit drops a watched pendingCount to 0 and when a subscription arrives while pendingCount is already 0 (otherwise no delta ever comes and the watcher never fires); it never fires synchronously and never through a bare microtask; on run, re-check and fire only if still 0, otherwise stay armed. The re-check is required because the zero-crossing happens inside one continuation of a promise whose later-registered consumers - suspended remainders of earlier-issued ops, already queued ahead of the verification - may install new promises. This is the counter form of "repeat scanning until settled": identical semantics, O(1) per iteration. On fire: errorCount > 0 → a single Error (sandbox collapse); full=true → deep copy through an old→new identity map consulted before recursing - one map serving two stated rules: visited-gate (termination) and identity-reuse (lookup-shared diamonds are DAGs and must not duplicate; COW deliberately does not preserve internal aliasing, normalize(full) deliberately does); otherwise return the already-marked branch. A non-extensible root never receives metadata or creates settlement waiters: validate its frozen subtree (pure, repeatable - frozen data is static), then treat it as already settled (0,0) - a violation returns the Error, full copies it plainly, otherwise it is returned as-is (already safe for COW).
hasError(root, segments) - resolve the parent path with the lookup machinery without shared ownership (pure inspection: only a boolean escapes, so nothing is marked), then inspect the final key: a broken path - intermediate Error, missing, or primitive - is true; a missing terminal property is false (no error there). At the reached branch it uses NO settlement waiters and answers for the current snapshot, counter-guided: errorCount > 0 → true immediately - safe, because a counted error sits under fully-settled positions, so no earlier-issued suspended remainder can still remove it (such a remainder would ride a promise on the error's path, which would make the error uncounted); pendingCount 0 → false immediately - every earlier remainder that could touch the branch rode a promise counted inside it, so the state is final with respect to all earlier operations; otherwise follow only the pending promises - descending only nodes with promiseCount > 0, so settled-clean subtrees are never entered - and probe again at each settlement, at hasError's FIFO slot, after the writeback has committed the resolved value's counts: first error wins (early exit), false once everything settles. A reached frozen branch is validated instead of probed. Imported Error values count as language errors.

Correctness:

Cycles: It is impossible to have cycles.
a.property.subproperty = a
The moment we read `a` through lookupPath it becomes shared. The moment we modify subproperty - `a` becomes copied (simpleAssignPath), thus it is a new object, no cycle. Import values will not be allowed to have cyclic data (either by requirement or actively validated on import). One back door remains: external code can resolve a promise with a language object it received earlier - boundary screening closes it by committing an Error instead of the back-edge. With subtree counters, acyclicity is load-bearing for delta-propagation termination, not merely for walk termination.

Object reuse: Any object that is reused - the runtime-visible ways this can happen are import and shared-ownership lookupPath, which mark the object as SHARED. An assignment and initialzier can target only one variable, so that the ownership is clear. var x = {a:1, b:2} - is owned by x and needs no shared mark. An async assignment to one variable is still owned by that one variable after the promise resolves. But var y = x will make the object shared for both x and y. External values enter through import, so their resolved data is marked shared through the same mechanism. A pure read or final ownership transfer uses sharedOwnership=false and therefore does not mark it.

Race conditions: with assignPath and deletePath, it may look as if we are opening the possibility for race conditions. While one operation awaits a promise that resolves some data, a subsequent operation may modify the data - or commit synchronously and overtake the suspended one entirely. Why neither can corrupt anything or diverge from sequential semantics:

Program order = FIFO order. Operations are issued in program order and register their continuations at their own program position; JS runs continuations on one promise in registration order. So all consumers of one promise mirror resume in program order, each advancing the mirror's `currentValue` on top of the previous operation's version (V → V′ → V″). Immediate resolution of all accessible promises is required for this to hold (use .then and keep walking, never await + enter).

Owned promise writeback does not mark by itself. Escape marking (shared-ownership lookupPath/import) and fork marking are what force advances to copy instead of mutating a value that another consumer or forked world still references.

Suspended reads are immune to overtaking writes: a suspended lookupPath continues through the mirror's private `currentValue`, never the live property. A later operation that synchronously reassigns or deletes the key (clearPromiseMirror + assignment) is invisible to it - the read resolves to the mirror value as of its own program position, exactly what sequential execution would have returned.

Suspended writes are fenced by the promise mirror map: a resumed write mirrors into the live tree only if its mirror still owns the key (map[key] === mirror - revoked by a later assignment or clearPromiseMirror). A revoked write completes on the disconnected mirror value - a wasted copy, never corruption - matching sequential semantics, where the later operation's assignment or delete overwrote that write anyway. In particular, a delete followed by a resumed write cannot recreate the deleted path.

Reassignment or delete at an ancestor orphans the whole subtree instead: local mirrors inside it may still succeed (the guard is deliberately local), but every effect is confined to the disconnected branch - a waste, not a validity issue - and the live tree matches sequential semantics, where the ancestor operation discarded that branch anyway.

COW forks of keys holding promises cannot reorder anything either: the copy's mirror is seeded from the source mirror's `currentValue` at the copier's own FIFO slot and the captured value is marked shared, so the copied world contains exactly the writes of operations issued before the copy, and operations issued after advance the two mirrors independently through copy-on-write (see Value resolve - Fork).
